# MCP

- 首次开源时间：2024-11-25
- 官方介绍：https://modelcontextprotocol.io/docs/getting-started/intro
- 说明：此条目用于记录 MCP 进入应用工程视野的时间节点与阶段意义，具体接入与使用方式应放在 Guide 中展开。

## 它是什么

MCP（Model Context Protocol）是一种为大模型连接外部工具、数据源与工作流提供统一接口的开放协议。

它试图解决的问题，不是“模型本身够不够强”，而是“AI 应用如何以更低摩擦的方式稳定接入外部能力”。

![image-20260416105313059](https://cdn.jsdelivr.net/gh/Qiuner/drawing-bed/2026/3/image-20260416105313059.png)

## 它把 AI 应用工程从哪一步推进到了哪一步

MCP 把 AI 应用工程从“每个产品各写各的工具接入层、协议层和适配器”，推进到了“围绕统一协议组织能力接入与复用”的阶段。

在 MCP 之前，工具调用往往是产品内部私有实现：每个客户端、每个 Agent 框架、每个工具服务端都要各自维护自己的接口约定。MCP 的出现，开始让“模型怎么接外部能力”这件事从零散工程问题转向标准化基础设施问题。

这也是它为什么值得被记录进时间轴：它不是单个产品能力更新，而是在试图重写 AI Agent 生态里的接口层。

## 现在看，它处于什么阶段

目前我把 MCP 标记为 `emerging`。

原因是它已经形成明显生态势能，并被越来越多客户端、开发工具和服务端接受；但距离真正成为跨平台、低摩擦、默认稳定的行业接口标准，还没有完全尘埃落定。

换句话说，MCP 已经不只是“一个官方提出的协议”，但它也还没有走到“像 HTTP 一样无需讨论”的阶段。

## 它可能替代什么

它可能替代一部分私有、碎片化、一次性工具接入层，让模型连接外部能力这件事从“每个团队各自实现”转向“围绕统一协议协作”。

如果这条路径成立，AI Agent 的外部能力接入会越来越像标准接口工程，而不是一堆难以复用的临时胶水代码。

## 它可能被什么替代

如果未来出现更轻量、更强约束、跨客户端兼容性更好的能力接入标准，MCP 也可能被更新一代的协议层取代。

尤其是在权限模型、身份认证、实时性和部署复杂度这些问题上，MCP 仍然有继续演化甚至被替换的空间。
